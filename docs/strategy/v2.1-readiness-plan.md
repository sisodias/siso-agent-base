# SISO V2.1 Readiness Plan

Status: readiness audit
Date: 2026-05-10
Current runtime generation: Gen0 / V1 iteration 104/109 boundary
Target upgrade meaning: V2 iteration 1, treated as `2.1.0` only after the gates below are true

Version boundary note: the user-framed stable baseline is iteration 104. At handoff time, local source metadata and installed runtime both report `0.1.109`. Treat `0.1.104` as the original pre-readiness baseline and `0.1.109` as the current pre-V2 source/runtime snapshot.

## Executive Summary

Do not rename SISO to V2 yet.

The current runtime is strong and heavily smoke-tested, but the repo still has a mix of active runtime capabilities, smoke-only capabilities, strategy docs, stale capability docs, and dormant scaffolds. Calling it V2 before reconciling those layers would make the version number a costume instead of a contract.

V2.1 should mean:

- SISO knows what it can do.
- SISO can prove what it did.
- SISO can run bounded fix loops without dragging the parent chat through every internal step.
- SISO keeps parent/child agent state scoped to the right chat/session.
- SISO has enforceable contracts, not just docs.
- SISO can measure efficiency gains against real tasks.

## The 10x Additions Worth Building

### 1. Checkpoint Autopilot

Highest ROI.

This is the feature most likely to turn 20 human prompts into 1 supervised command.

Design contract: `docs/strategy/autopilot-verifier-loop.md`

Current runtime slices:

- `siso action=check op=autopilot-plan`, validated by `npm run smoke:autopilot-plan`.
- `siso action=check op=fix|fix-loop|autopilot-fix`, validated by `npm run smoke:autopilot-fix-loop`.

Target behavior:

1. Take a scoped objective and validation command.
2. Create a checkpoint before edits.
3. Run the scoped check.
4. Extract a compact failure signature.
5. Use `rankedRepoMap`, `contextPack`, `publicCodeSearch`, and contracts to pick a narrow fix.
6. Patch only the likely files.
7. Rerun validation.
8. Stop when green, when max iterations are reached, or when failure signature stops changing.
9. Return one compact report to the parent chat.

V2.1 gate:

- Must prove at least a 5x reduction in parent-visible prompt turns on a fixed benchmark set.
- Must prove no raw logs/events flood parent details.
- Must checkpoint before file edits.
- Must never loop indefinitely.

### 2. Tree-Sitter Cached Repo Map

The current ranked repo map and Repo Index v1 are useful, but V2-grade code intelligence should be language-aware.

Target behavior:

- Parse JS/TS first, then shell/Markdown enough for this repo.
- Cache per file by path, mtime, size, and parser version.
- Track symbols, exports, imports, tests, docs links, and hotspots.
- Feed autopilot and tool selection.
- Report cache hit rate and map freshness.

V2.1 gate:

- Repo map generation should be faster on repeated runs.
- Ranked files should be more accurate than the current heuristic on benchmark tasks.

### 3. Contract Enforcement, Not Contract Documentation

Contracts should become a stop sign, not a suggestion.

Target behavior:

- Contracts have lifecycle: `draft -> active -> enforced -> retired`.
- `siso doctor contracts` maps changed files to required checks.
- Final answer/readiness gate refuses "done" claims when matched enforced contracts have not been verified.
- Contract report links capability IDs, test-space suites, and smoke commands.

V2.1 gate:

- At least these contracts are enforced:
  - release metadata
  - install/runtime
  - context safety
  - router/subagents
  - capability registry

### 4. Automatic Flight Recorder

Manual flight recording is useful, but V2 needs automatic evidence.

Target behavior:

- Every meaningful SISO run can emit a compact trace.
- Trace includes phase, tools, validations, child agents, contracts, source drift, duration, and compact risk notes.
- Parent-visible output stays small.
- Full evidence is stored on disk.

V2.1 gate:

- A normal `siso` task can produce a flight record without a special one-off recorder script.
- `analyze:flight-recorder` can compare runs and surface regressions.

### 5. Live Session-Scoped Event Bus And TUI

This directly addresses the user-visible bug class where subagents from one chat appear under another chat.

Target behavior:

- Every event has `sessionId`, `threadId`, `parentRunId`, `childRunId`, and `visibility`.
- UI subscribes only to events for the active thread/session.
- Global status can show global health, but child-agent rows are session-scoped by default.
- Completed child notifications re-enter only the owning parent, never every chat.

V2.1 gate:

- A smoke creates two sessions, starts child work in one, and proves the other session shows none of those child rows and receives no child completion payload.

### 6. Persistent Executive MVP

This is a strategic multiplier, but it should not be allowed to become vague autonomy.

Target behavior:

- Manual `.siso/executive/` state remains the source of truth first.
- Add narrow commands:
  - `siso executive status`
  - `siso executive capture`
  - `siso executive review`
  - `siso executive plan`
- Every executive action cites a goal, decision, task, or inbox item.

V2.1 gate:

- A future agent can run one command and know current goals, active tasks, next actions, and stale decisions.

### 7. Parent/Child Cost And Efficiency Scorecard

We need to prove "10x", not feel it.

Target behavior:

- Roll up parent and child turns, prompt/output tokens when available, event counts, run duration, tool calls, validation attempts, and retry count.
- Store as benchmark results and flight-recorder metrics.
- Compare installed baseline `0.1.104` and source/runtime snapshot `0.1.109` to the V2 candidate on the same tasks.

V2.1 gate:

- Publish a before/after scorecard for at least 10 representative tasks.
- Include parent-visible tokens/characters, prompts, elapsed time, validation passes, and manual intervention count.

### 8. Capability Truth Reconciliation

The repo should stop having "implemented" in one file and "idea" in another.

Target behavior:

- `docs/capabilities/current.md`, `ideas.md`, and `changelog-candidates.md` match the registry.
- Stale research docs get a top note saying what has since shipped and what remains open.
- Capability audit checks likely missing registry/docs entries by comparing package scripts, files, and changelog.

V2.1 gate:

- No known shipped feature is listed as merely an idea.
- No V2 claim depends on a dormant scaffold.

## V2.1 Validation Contract

Do not ship V2.1 until these checks are true:

- `npm run smoke:all` passes.
- `./scripts/install-local.sh` passes.
- `siso doctor` passes.
- `siso where` reports source/runtime match.
- Contract diff reports no unenforced required checks for changed files.
- Source drift reports zero errors and zero warnings, or documented warnings.
- Tool-selection eval passes and is part of the package script surface.
- Two-session child-agent isolation smoke passes.
- Autopilot benchmark passes bounded-loop safety checks.
- Flight-recorder analysis can compare baseline and candidate.
- V2 readiness doc has been updated with the exact release evidence.
- Agent handoff brief has been updated when architecture, active lanes, or next task changed.

## Recommended Upgrade Sequence

1. Documentation truth reset.
   - Update stale capability docs.
   - Mark shipped research items as shipped.
   - Mark dormant scaffolds as dormant or deferred.

2. Enforcement layer.
   - Contracts lifecycle.
   - Final-answer readiness gate.
   - Source drift plus contract gate in doctor/readiness.

3. Measurement layer.
   - Automatic flight recorder.
   - Cost/session rollup.
   - V2 scorecard.

4. Autopilot.
   - Checkpoint.
   - Run.
   - Failure signature.
   - Patch.
   - Rerun.
   - Compact result.

5. Code intelligence.
   - Tree-sitter cached repo map.
   - Public code search stays available as outside-pattern lookup.

6. Operator experience.
   - Live event bus.
   - Session-scoped TUI.
   - Executive status/review commands if included in V2.1.

## Decision

The next biggest, highest-ROI feature is checkpoint autopilot, but it should be built on top of:

- enforced contracts,
- automatic flight recording,
- session-scoped events,
- and the ranked repo map that already shipped.

That combination is the actual 10x path: not just one clever tool, but a bounded loop that can inspect, edit, verify, measure, and report without flooding the parent chat or leaking child-agent state across sessions.
